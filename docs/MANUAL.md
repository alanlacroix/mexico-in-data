# The Mexico Brief — operating manual

The one document an agent reads first. It indexes and arbitrates the laws; it does not duplicate them. Locked from the scaffolding audit plan (Fable, 2026-07-12) + Alan's decisions. Internal, never deployed.

## 0. Precedence

This manual > the named laws below > everything else. **Any doc not named here is historical** (see `docs/archive/`). When two rules seem to conflict, this manual decides.

## 1. What this site is

A free, public, trust-first briefing on Mexico, built by Alan for himself first and open to all. Every figure is official, dated, and one click from its source. If we cannot validate it, we remove it; we never fabricate. Static Eleventy site, Cloudflare Pages, deploy = push `main`. The repo *is* the database: every number's provenance is a commit.

## 2. Data

**Storage.** Site data = flat JSON committed to git (`data/`), regenerated by scheduled jobs. This is the ledger — versioned, auditable, free. **Supabase is private and optional** (captured news + a revision history); the site never reads from it. Subscribers live in Beehiiv. Do not move site data into a database, and do not add one.

**Cadence** (codifies what already runs; the governing rule: *insights never refresh more often than the numbers under them, and never before them*):

| What | Cadence | Who writes it |
|---|---|---|
| Numbers (macro, trade, fuel, news wire) | every ~6 hours | deterministic code, no model |
| Insights (homepage brief, "happening", area blurbs) | ~4×/day, after the numbers | model, reads finished numbers |
| The Read (weekly analysis) | Monday draft | model drafts, Alan publishes |
| Daily email (future; not shipped) | no delivery workflow | deterministic, no model |
| Weekly email (analysis) | Monday morning, hand-sent | model drafts, Alan approves |
| Standing-context story block | monthly | model drafts, Alan approves |

**Staleness.** A failed pull keeps the last-good value (never a blank). Every tile shows its observation period. If a source is > 2 cycles late, its stamp turns amber; if > 30 days, the tile says so in words. Never show an old number as fresh.

## 3. Structure

**One section template, six slots, every topic page identical, no exceptions:**

`Hero → Story → Board → Background → Watch → Go Deeper`

- **Hero** — H1 (short label) + one-sentence dek (one checkable dated fact).
- **Story** — the standing-context block (§8), above the numbers.
- **Board** — 3–6 live number tiles, `repeat(3,1fr)`, no blank trailing cell. NEXT = one line on the board header.
- **Background** — exhibits, from the chart vocabulary in `visual-grammar.md`.
- **Watch** — the forward calendar of record.
- **Go Deeper** — sources + related pages.

Pages hold **no layout code, no CSS, no page-local functions** — each page is a config (which series feed the tiles, which exhibits, the story text, the watch items). The shared library (`assets/mb.js` + one stylesheet) renders it. If a page needs something the library lacks, the component goes into the library and this manual, or it does not ship.

**Adding data** = write `data/series/<id>.json` + one config line. A whole new page = one config file. That is the whole point: add data, never rebuild.

**New-metric intake checklist:** source · license · cadence · connector · series file · page slot.

**The homepage is not this template — it has its own shape (Fable 2026-07-12).** Order, and the DOM order equals the mobile reading order (desktop lifts the rail top-right via `grid-template-areas`, never CSS `order`, so screen-reader order matches): **Lead (H1) → Brief → What stood out → Numbers → Next up → Topics.** Understand the present before the schedule.
- **Brief** = the whole week, 3–5 rubric-ranked stories, all visible — never truncated (the briefing IS the product; hiding stories makes it a feed). On mobile the tails compress: tighter spacing, ranks 4–5 go headline-only. H1 = story #1 as a label; the Brief's first paragraph explains it, so the lead is never printed twice.
- **Next up** = a quiet dated-catalyst rail, a margin note not a second column (date/title/source only; no mechanism/chips/kind-bars on the compact view). It shows the next *distinct* things: any event in the **lead story's topic is suppressed** from the compact rail, so the headline never echoes a third time. **Guard: when an event is suppressed, its date must appear in the lead block** — the H1 carries the story but not the date (a pre-publish check in the weekly review). "All N dates →" inline-expands to the full calendar with the why on each. A standalone calendar page waits until events regularly exceed ~8–10.

What/Now/Next is dead. "Latest developments" (dated news) never appears on a topic page — news lives only in the Brief and The Read (§6).

## 6. The LLM boundary — a closed list of five

The model runs in exactly five editorial places: **happening · homepage brief · the area blurbs · the email · The Read.** Every one only *arranges* finished numbers; none produces a number; each falls back to deterministic output if the API key is absent. **The number path (source API → connector → validation → JSON → site) is 100% model-free.** Adding a sixth LLM touchpoint requires a Fable ruling.

## 7. Email — two products, one gate that matters

- **The Daily** — a future product, not shipped and not promised on the site. If built, it is pure data: the peso's official close, any release that landed, today's scheduled releases, and the top ~3 curated headlines (title + source + link, no summary). Deterministic, no model.
- **The Weekly** — Monday morning, the flagship: the week's changes + analysis + standing context, every figure linked. Model drafts it Saturday; **Alan reads it, prepares the Beehiiv handoff, checks a Beehiiv test, and sends from Beehiiv.** Nothing sends on a schedule.

Beehiiv is the canonical subscriber list and the only delivery system. `prepare-beehiiv-review` only exports a review artifact and has no provider secret. The manual Post Builder step is intentional while Beehiiv's Send API is Enterprise-only.

Site copy describes only what ships today (the weekly). When the daily is built, the copy adds it. Marketing never promises a cadence the machinery does not deliver.

## 8. The standing-context "story" block

A slow-changing context piece at the **top** of each section page, above the numbers (SHIPPED on all 6 pages, 2026-07-12). Model drafts it, **Alan approves** before it shows. **Drift rule (non-negotiable — a stale literal two inches above the live tiles is a trust kill):** the story carries ONLY slow, structural, dated numbers (export shares, a two-thirds supermajority, an annual poverty rate). It NEVER states a fast board snapshot (spot FX, the latest monthly print, the current policy rate); the board right below carries those, live. Fix a fast number by describing the structure instead, or by computing it from the live series — never by hand-typing this month's value. Same honesty rules as everything else.

## 9. Enforcement (Phase 2, to build)

The build fails, not a checklist. `pipeline/assert.js`: (1) title/period ≠ plotted endpoint; (2) a rate series shown with relative % instead of `pp`; (3) displayed total ≠ its components at shown precision; (4) a hard-coded count ≠ the loaded registry; (5) pinned-prose drift (`pinned.json`, warn). Plus a page lint: no inline `<style>`, no page-local render functions, template slot order intact. Plus the mexico-copy LINT for prose the assertions can't see.

## 10. Ownership

Alan approves content (email, The Read, the story block) and product forks. Fable owns rulings (structure, the LLM boundary, cadence changes). Opus/agents execute to this manual.
